# 需求拆分：桌面 AI Copilot

> 基于 `initial_intention.md` 的初步需求拆分，用于讨论和对齐方向。

## 一、核心模块拆分

### 1. 屏幕感知层（Perception）
负责从操作系统采集用户活动的原始信号。

- **屏幕截图采集**：采样频率、分辨率压缩、是否多屏支持
- **活动窗口识别**：当前前台 App、窗口标题、进程信息
- **输入信号采集**：鼠标点击/移动、键盘输入（高敏感）、滚动事件
- **OCR / UI 元素提取**：从截图中提取文本和可交互控件
- **音频信号**（可选）：是否需要采集系统音频或麦克风

### 2. 行为理解层（Understanding）
把原始信号聚合为有语义的"事件"和"任务"。

- **短期行为识别**：用户此刻在做什么（看代码 / 写邮件 / 看视频 / 浏览网页）
- **上下文聚合**：把多帧截图 + 操作序列拼成连续事件
- **意图推断**：用户的目标是什么（调 bug / 写报告 / 学习新概念）
- **任务边界检测**：何时算"开始一个任务"，何时算"切换"
- **状态管理**：维护一个"用户当前任务栈"

### 3. 大模型决策层（Reasoning）
负责调用 LLM 并决定何时、向谁、用什么方式提建议。

- **模型选择**：云端大模型 vs 本地小模型 vs 混合架构
- **Prompt 构造**：截图 + 文本上下文 + 历史行为 + 用户画像
- **触发策略**：何时调用大模型（不能每秒都问，成本和延迟都不行）
- **建议生成**：什么时候该弹出，什么时候保持沉默
- **缓存与去重**：避免短时间内重复建议

### 4. 交互呈现层（Presentation）
建议如何呈现给用户，且不打扰。

- **呈现形式**：浮窗 / 侧边栏 / 系统通知 / 快捷键召唤
- **打扰策略**：用户专注时不能打断（专注度评估）
- **反馈闭环**：用户接受/忽略/拒绝后如何学习偏好
- **召唤式 vs 主动式**：是否同时支持用户主动提问

### 5. 隐私与安全层（Privacy）
这一层的优先级**非常高**，做不好整个产品没人敢用。

- **本地 vs 上传**：哪些数据绝对不能离开本机
- **敏感内容过滤**：密码框、银行页面、私人聊天、IDE 中的密钥
- **白名单 / 黑名单**：用户可指定哪些 App / 网站 启用或排除
- **数据存留策略**：截图和行为日志保留多久、加密存储
- **可观测性**：用户能随时看到系统在采集什么、上传了什么

## 二、决定架构走向的关键问题

在继续往下细化之前，需要先对齐下面 5 个问题：

### Q1. 目标用户和场景
- 程序员（编码助手）
- 办公人群（效率助手）
- 学生（学习伴侣）
- 通用人群

> 这决定了"理解行为"的难度天花板，以及是否需要垂直领域知识。

### Q2. 优先平台
- Windows / macOS / Linux

> 三个平台的屏幕捕获 API、权限模型、打包发布方式差别很大。建议先选一个。

### Q3. 大模型部署形态
- 纯云端（便宜快、隐私差）
- 纯本地 VLM（隐私好、硬件门槛高、效果差一截）
- 混合（本地做粗筛 + 云端做深度理解）

### Q4. "主动式"的边界
- 定时主动弹出（容易被嫌烦）
- 检测到"用户卡住"才弹（难度高，但体验好）
- 完全召唤式 + 偶尔主动（折中方案）

### Q5. MVP 范围
- 完整框架先搭起来（周期长，风险大）
- 单一垂直场景跑通闭环（建议方案）

## 三、MVP 推荐方案（待讨论）

**单一场景 + 单平台 + 云端模型**，把闭环跑通后再扩展。

理由：
- 屏幕捕获本身在不同平台/不同 App 下的兼容性陷阱很多，先聚焦一个平台
- 云端模型能力强、迭代快，先用云端验证"行为理解 → 建议"链路是否真的有用
- 单一场景能让"理解准确度"和"建议有用度"两个指标更容易评估

**待选垂直场景示例**：
1. **看代码时给解释 / 找文档**：场景明确，目标用户（程序员）需求强
2. **看英文资料时给翻译 / 摘要**：触发清晰，价值直观
3. **写文档时给润色 / 续写建议**：依赖 IDE 集成的话场景受限
